# Feature Specification: File Upload Redesign

## User Scenarios & Testing

### User Story 1: Inline File Upload with Progress
**Priority:** P1
**As a** user, **I want** to upload files through an inline upload zone in the document list (rather than a modal), **so that** I don't lose context while uploading and can see progress directly.

#### Acceptance Scenarios
- **Given** I am on the document list page, **When** I drag a file onto the inline upload zone, **Then** the file begins uploading and a progress bar appears inline.
- **Given** an upload is in progress, **When** I continue browsing the document list, **Then** the progress indicator remains visible without blocking my interaction.
- **Given** an upload fails, **When** I view the upload zone, **Then** I see a retry button next to the failed file.

#### Edge Cases
- User navigates away from the page during upload — upload continues in background if supported, or shows a warning prompt.
- Browser tab is closed mid-upload — handled by resume-on-return (see User Story 4).

### User Story 2: Large File Upload (up to 500MB)
**Priority:** P1
**As a** enterprise user, **I want** to upload files up to 500MB, **so that** I can share large documents, datasets, and media files without workarounds.

#### Acceptance Scenarios
- **Given** I select a 250MB file, **When** the upload begins, **Then** it is transmitted in 5MB chunks and completes successfully.
- **Given** I select a 501MB file, **When** I attempt to upload, **Then** I see a clear error message stating the 500MB limit.

#### Edge Cases
- Network interruption during a chunk upload — the system should retry the failed chunk automatically.
- Server storage quota exceeded — display a meaningful error, not a generic failure.

### User Story 3: Concurrent File Uploads
**Priority:** P1
**As a** user, **I want** to upload multiple files simultaneously (up to 3 at once), **so that** I can batch my uploads and save time.

#### Acceptance Scenarios
- **Given** I drag 3 files onto the upload zone, **When** the uploads begin, **Then** I see 3 separate progress bars, each tracking independently.
- **Given** 3 uploads are in progress and I try to add a 4th, **When** I drop the file, **Then** it is queued and starts automatically when a slot opens.

#### Edge Cases
- One of the concurrent uploads fails — it should not affect the other active uploads.

### User Story 4: Resume Failed or Interrupted Uploads
**Priority:** P2
**As a** user, **I want** to resume an upload that was interrupted (by network failure or browser close), **so that** I don't have to restart large uploads from scratch.

#### Acceptance Scenarios
- **Given** an upload was interrupted at 60% completion, **When** I return to the document list, **Then** I see a "Resume Upload" option for the interrupted file.
- **Given** I click "Resume Upload", **When** the upload resumes, **Then** it continues from the last successfully uploaded chunk, not from the beginning.

#### Edge Cases
- Staged chunks expire on the server before the user returns — show "Upload expired, please re-upload" message.
- Multiple interrupted uploads exist — show resume option for each.

## Requirements

### Functional Requirements
- **FR-001:** The system shall replace the current modal upload UI with an inline upload zone embedded in the document list view (Option B design).
- **FR-002:** The system shall use the tus resumable upload protocol for all file uploads.
- **FR-003:** Files shall be uploaded in 5MB chunks.
- **FR-004:** The maximum file size shall be 500MB.
- **FR-005:** The upload zone shall display a progress bar for each active upload.
- **FR-006:** The system shall support up to 3 concurrent file uploads, with additional files queued automatically.
- **FR-007:** A temporary staging area (Azure staging container) shall hold chunks during upload before assembly.
- **FR-008:** All uploaded files shall undergo virus scanning after chunk assembly and before being moved to final storage.
- **FR-009:** The system shall support resuming interrupted uploads by detecting previously staged chunks when the user returns.
- **FR-010:** File type restrictions shall remain unchanged (all types except executables).

### Key Entities
- **UploadSession:** A tus-protocol upload session tracking chunks, progress, and status for a single file
- **StagingContainer:** Azure blob storage container for temporary chunk assembly
- **VirusScanPipeline:** The security scanning process applied to assembled files before final storage
- **UploadQueue:** Client-side queue managing concurrent upload slots (max 3)

## Success Criteria
- Users can upload files up to 500MB without failure
- Upload progress is visible inline without modal interruption
- Interrupted uploads can be resumed from the last completed chunk
- All files pass virus scanning before reaching final storage
- Concurrent upload of 3 files works without performance degradation

## Assumptions
- The tus protocol library is available and compatible with the current frontend framework
- Azure staging container can be provisioned within the existing infrastructure budget
- The virus scanning service is already available (referenced as a security team requirement)
- Current file type restrictions (no executables) are sufficient and won't change for this feature
